## 1. 端口与接口设计

- **端口**：`8000`（可用环境变量 `PORT` 改；`HOST` 改监听地址）
- **基地址（baseURL）**：`http://127.0.0.1:8000`
- **返回格式**：全部 JSON；已开启 CORS（前端任意端口可直接 fetch）
- **约定参数**：
  - `case_id` ∈ `buyer1_clean` / `buyer1_attack_type1` / `buyer1_attack_type2` / `buyer2_clean` / `buyer2_attack_type1`
  - `skill_id` 用 `code_review`
  - `stage` ∈ 见第 2 节八步标识

### 接口清单（15 个）

| 用途 | 方法 路径 | 前端什么时候用 |
|---|---|---|
| 健康检查 | `GET /api/health` | 启动时探活 |
| **界面文案** | `GET /api/ui/text` | 启动时拉取，填充所有界面静态文字（见下「界面文案」一节） |
| 全局配置 | `GET /api/demo/meta` | 拿默认 skill、买家列表、攻击族、八步元信息 |
| **案例列表** | `GET /api/demo/cases` | 渲染「②Case 选择台」5 张卡片 |
| **★案例全流程** | `GET /api/demo/cases/{case_id}` | 渲染「③故事流」整页（最重要） |
| 单个步骤 | `GET /api/demo/cases/{case_id}/steps/{stage}` | 点开某一步要细节时（可选） |
| **逐步播放(SSE)** | `GET /api/demo/cases/{case_id}/stream?delay=0.9` | 想要「运行中」动画时 |
| 原始 Skill | `GET /api/skills/{skill_id}/raw` | 第 1 步看原文 |
| 水印 Skill | `GET /api/skills/{skill_id}/watermarked` | 三栏对比时 |
| SkillIR 结构 | `GET /api/skills/{skill_id}/ir` | 第 2 步节点图（也已内嵌在全流程里） |
| 买家列表 | `GET /api/skills/{skill_id}/buyers` | 下拉选买家 |
| **买家指纹** | `GET /api/watermarks/{skill_id}/buyers/{buyer_id}` | 第 4 步 32 位指纹格子 |
| **指纹对比** | `GET /api/watermarks/{skill_id}/compare?a=buyer_1&b=buyer_2` | buyer_1 vs buyer_2 逐位 diff |
| 探针查询 | `GET /api/queries/{skill_id}?kind=all\|positive\|negative\|normal` | 想单独看探针文本 |
| **指标总览** | `GET /api/metrics/overview` | 渲染「⑤指标总览页」 |

> **前端 90% 的工作只用两个接口**：`/api/demo/cases`（列表）+ `/api/demo/cases/{case_id}`（全流程）。其余都是锦上添花。

## 1.5 界面文案：`GET /api/ui/text`

所有界面静态中文文字（导航、背景介绍、各区块标题、按钮、8 个 visual 渲染器里的内联标签、裁决卡、指标表头等）都集中在这个接口下发，**前端不需要再内嵌任何业务文字**。启动时拉一次即可。

返回是一个按区域分组的字典（`meta` / `offline` / `hero` / `sections` / `controls` / `footer` / `phases` / `attack_tags` / `case_card` / `story` / `src_tag` / `visual` / `verdict` / `metrics` / `bit_tooltip`）。其中 `visual` 下再按 8 种 `visual.type` 分组（`skill_document` / `node_graph` / …）。

**占位符约定**：需要插入运行时变量的句子用 `{name}` 占位符，并保留 HTML 标记。例如：

```jsonc
"visual": {
  "bit_grid": {
    "intro": "买家 <b>{buyer_id}</b> 的指纹 = 一串 <b>{codeword_length} 位</b>纠错码（编码规则：{bit_rule}）"
  }
}
```

前端拿到模板后把 `{buyer_id}` 等替换为对应数据即可（需转义的值先做 HTML 转义）。参考实现见 `frontend/app.js` 里的 `fmt()` 与 `applyStaticText()`。

> 唯一仍留在前端的文字是「后端未连接」时的兜底提示——因为那一刻后端正好不可达，无法由接口下发；其同样的文案在本接口的 `meta.conn` / `offline` 里也有一份。

## 2. 核心数据结构：`GET /api/demo/cases/{case_id}`

这是整个网站的主数据源。返回结构：

```jsonc
{
  "case_id": "buyer1_attack_type2",
  "title": "Buyer 1 · 第二类攻击",
  "skill_id": "code_review",
  "buyer_id": "buyer_1",
  "attack": { "id": "...", "name": "...", "description": "...", "text_effect": "..." },
  "goal": "这个案例想证明什么（一句话）",

  "provenance": {                     // 数据出处，建议页脚展示，体现“真实数据”
    "data_source": "real",            // real=真实模型输出 / simulated=回退推导
    "model": "openai/gpt-5-mini",
    "agent": "LangChain",
    "run_tag": "gpt5mini_langchain",
    "note": "步骤6–8均来自真实模型输出，并与官方指标交叉印证。"
  },

  "summary": {                        // 结论卡（④结果裁决页）直接用这个
    "ownership": "verified",
    "attributed_buyer": "buyer_1",
    "attribution_correct": true,
    "true_ws": 1.0, "false_ws": 0.0187, "margin": 0.9813, "score_own": 0.9813,
    "confidence": 1.0, "errors": 0, "erasures": 0,
    "decode_margin": 16, "ecc_satisfied": true, "data_source": "real"
  },

  "timeline": [ /* 8 个步骤，见下 */ ]
}
```

每个 `timeline[i]` 步骤统一是这个形状：

```jsonc
{
  "stage": "ownership_scored",        // 步骤稳定标识
  "index": 7,                         // 第几步
  "title": "胶囊提取与所有权评分",      // 大标题
  "phase": "verification",            // embedding/deploy/verification 三大阶段，可用于配色
  "story": "主线一句话（大字，给评委念的）",
  "technical_hint": "技术细节（点开折叠区才显示，含论文公式）",
  "paper_ref": "§3.4 Step 2，式(8)(9)",
  "visual": { "type": "score_bar", /* 该步专属可视化数据 */ }
}
```

**前端渲染逻辑就一句话**：循环 `timeline`，每步固定渲染「标题 + story 大字 + 折叠的 technical_hint/paper_ref」，再根据 `visual.type` 选一个组件渲染 `visual`。

### 八步 → visual.type → 建议组件

| # | stage | visual.type | 该步关键字段（visual 里） | 建议前端组件 |
|---|---|---|---|---|
| 1 | `skill_loaded` | `skill_document` | `sections[]`(标题/类型/正文)、`markdown_excerpt` | Markdown 卡 + 章节高亮 |
| 2 | `skillir_built` | `node_graph` | `nodes[]`(type/label/text)、`edges[]` | 节点图（六类节点配色） |
| 3 | `owner_watermark_embedded` | `capsule_schema` | `auxiliary_clauses[]`、`capsule_fields[]`、`example_capsule` | 条款列表 + 五字段表 + 胶囊代码块 |
| 4 | `buyer_fingerprint_embedded` | `bit_grid` | `fingerprint_bits`(32字符)、`positions[]`、`compare_with` | 32 格 0/1 网格 + 对照买家 diff |
| 5 | `attack_applied` | `attack_diff` | `name`/`text_effect`、`true_bits`/`observed_bits`、`affected_positions` | 前后对比 + 受影响位高亮 |
| 6 | `differential_probing` | `probe_pair` | `positive_example`/`negative_example`(query+capsule_output) | 左右两栏：正探针吐胶囊 / 负探针正常回答 |
| 7 | `ownership_scored` | `score_bar` | `true_ws`/`false_ws`/`margin`/`threshold`/`official` | 双分数条 + 阈值线 +「官方对照」小字 |
| 8 | `buyer_decoded` | `decode_grid` | `observed_bits`、`ranking[]`(buyer/errors)、`top1`/`top3`/`decode_margin`/`official` | 排名表（top1 距离0、其余≥16） |

> 小贴士：步骤 6–8 的 `visual` 里都有 `data_source`，步骤 7/8 还有 `official` 字段。
> 想强调「真实数据 + 论文一致」就把 `official` 的数字并排显示（例：实测 True-WS 1.0 / 官方 0.997）。

## 3. 页面设计（5 个区块，从上到下）

整站建议做成**单页滚动叙事**（也可拆 5 个路由）。每个区块对应的接口与组件如下：

### ① 首页 / 知识背景
- **数据**：静态文案（可硬编码，或放一份 `intro.json`）。
- **内容**：Skill 是什么、为什么易被盗、两个技术难点（scale/readability）、我们的答案。
- **目的**：30 秒让评委明白「这是个真问题」。

### ② Case 选择台
- **接口**：`GET /api/demo/cases`
- **渲染**：5 张卡片，显示 `title` / `goal` / `attack`。点击进入③。
- **加分**：卡片上写明两类攻击区别（第一类=盲改写，第二类=定向删条款）。

### ③ 故事流时间线（核心页）
- **接口**：`GET /api/demo/cases/{case_id}`（一次拿全 8 步）
- **布局**：左侧/顶部一条 8 节点时间轴；主区域按当前步渲染「story 大字 + visual 组件 + 折叠技术细节」。
- **交互**：上一步/下一步切换；或点「播放」走 SSE（见下）。
- **配色**：用 `phase` 区分三大阶段——`embedding`（嵌入·蓝）/`deploy`（部署·灰）/`verification`（验证·绿）。
- **播放动画（可选）**：`GET /api/demo/cases/{case_id}/stream?delay=0.9`
  - 这是 SSE。前端用 `EventSource` 监听，事件名就是 `stage`（`skill_loaded`…`buyer_decoded`），最后一个事件 `report_ready` 带 summary。每收到一个事件点亮一步即可。

### ④ 结果裁决页（每个案例收尾）
- **数据**：直接用③返回里的 `summary` + `provenance`。
- **渲染**：一张「判决书」卡：
  - ✅ 所有权：`ownership` + `true_ws`/`false_ws`/`margin`
  - 🎯 溯源买家：`attributed_buyer` + `confidence` + `errors`/`erasures` + `ecc_satisfied`
  - 🛡 攻击类型：`attack.name`
  - 📊 数据出处：`provenance`（真实模型 + 官方指标一致）
- **5 个案例对比点**（卡片底部一句话）：基线干净命中 / 改写后仍命中 / 删条款最难仍命中 / 换买家精准区分 / 换买家+攻击仍稳。

### ⑤ 实验指标总览页
- **接口**：`GET /api/metrics/overview`
- **渲染两块**：
  - `paper_aggregate`：论文整体数字（所有权 99.2%、买家 Top-1 99.3%、鲁棒性、保真、消融）→ 我们 vs 基线对比条形图。
  - `run_official`：本次真实运行的逐域实测指标 → 表格（证明「不仅论文说有效，我们这次也实测有效」）。

## 4. 前端最小实现路径（建议顺序）

1. 写个 `api.js` 封装 baseURL + fetch。
2. 做②列表页：`GET /api/demo/cases` → 5 张卡。
3. 做③详情页：`GET /api/demo/cases/{case_id}` → 循环 `timeline`，先用「标题+story+JSON.stringify(visual)」占位跑通。
4. 给 8 种 `visual.type` 各写一个组件，逐个替换占位。
5. 做④结论卡（用 summary）、⑤指标页。
6. 最后加 SSE 播放、折叠技术细节、配色动画等润色。

> 只要第 2、3 步做完，整个故事就能从头讲到尾了。其余都是体验增强。
